# 代码分层与目录职责

## 1. 目的

本项目采用分层组织方式。为了避免模块职责混乱，需要明确不同目录的边界，保证：

- 新代码能快速判断放在哪一层
- 同类逻辑尽量集中
- 避免 `utils`、`services`、`adapters` 等目录变成兜底杂项目录

---

## 2. 总体原则

### 2.1 优先放在语义最明确的一层

如果某段代码同时“看起来也能放进别的目录”，优先选择职责最收敛、最明确的目录。

### 2.2 底层模块不依赖上层业务流程

例如：

- `utils` 不应依赖 `services`
- `repositories` 不应依赖 `workflows`
- `adapters` 不应直接承接业务流程编排

### 2.3 公共能力与业务能力分开

- 公共工具放 `utils`
- 业务动作放 `services`
- 决策规则放 `strategy_engine`
- 完整流程放 `workflows`

---

## 3. 目录职责

### 3.1 `src/utils`

放通用基础工具和基础设施封装。

适合放：

- 配置读取
- 日志工具
- 数据库连接池
- 通用 HTTP 请求封装
- 报告周期/日期解析
- 与具体业务弱相关的纯工具函数

不适合放：

- Amazon 广告业务规则
- 拓词流程编排
- 关键词分类决策

判断标准：
如果一段代码去掉当前业务背景，仍然可以作为基础能力复用，优先放 `utils`。

---

### 3.2 `src/adapters`

放外部协议、外部输入输出格式与内部结构之间的适配逻辑。

适合放：

- LLM 消息格式转换
- 图片编码/解码转换
- 外部接口字段到内部结构的映射

不适合放：

- 直接发起外部请求
- 业务流程判断
- 数据库访问

判断标准：
如果代码的核心工作是“转换格式”或“做协议层适配”，放 `adapters`。

---

### 3.3 `src/apis`

放对外部服务的直接调用入口。

适合放：

- 调用本地或远程 LLM 服务
- 调用三方 HTTP API
- 封装请求参数、超时、响应读取

不适合放：

- 复杂业务编排
- 业务规则决策
- 数据库存取

判断标准：
如果代码的核心工作是“请求外部服务并返回结果”，放 `apis`。

---

### 3.4 `src/repositories`

放数据库和原始报表的数据访问层。

适合放：

- SQL 查询
- 原始行数据读取
- 面向表结构的访问封装

不适合放：

- 指标计算
- 业务规则判断
- 工作流编排

判断标准：
如果代码的核心工作是“从数据库/原始数据源拿数据”，放 `repositories`。

---

### 3.5 `src/data_processor`

放纯数据计算、纯指标处理和纯转换逻辑。

适合放：

- 时间加权
- KPI 计算
- 趋势计算
- 份额归一化
- 特征构造

不适合放：

- 查库
- 调外部 API
- 业务流程判断

判断标准：
如果函数只依赖输入数据，输出计算结果，不依赖流程上下文，放 `data_processor`。

---

### 3.6 `src/services`

放业务动作、业务用例和模块组合逻辑。

适合放：

- 组合 `repositories` 和 `data_processor`
- 根据业务规则筛选数据
- 汇总候选词
- 组织某一业务步骤的输入输出

不适合放：

- 完整工作流编排
- 通用基础工具
- 单纯 SQL 查询

判断标准：
如果代码表达的是“业务上要做什么”，并且会组合多个下层模块，放 `services`。

---

### 3.7 `src/strategy_engine`

放策略规则、决策逻辑和规则引擎。

适合放：

- 位置策略
- bid 策略
- 预算策略
- 拓词强度策略
- 关键词分类决策

不适合放：

- 查库
- 外部 API 请求
- 完整流程串联

判断标准：
如果代码的核心工作是“根据规则做决策”，放 `strategy_engine`。

---

### 3.8 `src/workflows`

放端到端业务流程编排。

适合放：

- 搜索词精准投放拓词主流程
- 多步骤业务链路的顺序组织
- 跨多个 service / strategy 的流程控制

不适合放：

- 底层工具实现
- 纯规则实现
- 单个数据访问函数

判断标准：
如果代码描述的是“这个业务整体怎么从输入走到输出”，放 `workflows`。

---

### 3.9 `src/schemas`

放输入输出数据结构、内部传递对象定义。

适合放：

- Workflow request/response
- Strategy result
- Candidate / Seed 对象

不适合放：

- 业务流程逻辑
- 数据计算逻辑

---

### 3.10 `src/constants`

放固定常量、枚举和静态配置。

适合放：

- 指标阈值
- 广告位常量
- 拓词强度配置

不适合放：

- 依赖运行态上下文的逻辑
- 复杂业务处理

---

## 5. 判断放置位置的快速问题

当一段代码不知道该放哪时，可以按下面顺序判断：

1. 它是不是纯 SQL / 原始数据读取？
   是：放 `repositories`

2. 它是不是纯计算，不依赖业务流程？
   是：放 `data_processor`

3. 它是不是外部协议或格式转换？
   是：放 `adapters`

4. 它是不是直接请求外部服务？
   是：放 `apis`

5. 它是不是业务动作或模块组合？
   是：放 `services`

6. 它是不是规则和决策？
   是：放 `strategy_engine`

7. 它是不是端到端流程编排？
   是：放 `workflows`

8. 它是不是基础设施或通用工具？
   是：放 `utils`

---

## 6. 后续维护建议

- 新增目录前，先确认现有目录是否已经能表达该职责
- 避免出现多个语义接近的目录同时承载同类逻辑
- `utils` 和 `services` 是最容易膨胀的两个目录，必须严格控制边界
- 如果某个工具明显只服务于“报告周期语义”，优先放 `utils/report_period.py`，不要塞进泛化工具文件
